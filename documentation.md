### 5.4 Требования к разработке сервиса проверки уникальности объекта интеллектуальной собственности в национальном мессенджере

- Сервис проверки уникальности объекта интеллектуальной собственности (далее - сервис) должен быть реализован в виде
  интеграции с национальным мессенджером (далее - мессенджер) и соответствующего интерфейса мессенджера, обеспечивающих
  возможность направления запросов и получения результатов поиска из Системы в режиме реального времени непосредственно
  в
  интерфейсе мессенджера.
- Сервис должен функционировать, предоставляя пользователям мессенджера доступ к функциональности Системы без
  необходимости авторизации в веб-интерфейсе ИС «Поисковая платформа».
  Все компоненты необходимые для полноценного функционирования сервиса в целевой среде эксплуатации (включая
  интеграционное взаимодействие с заявленными внешними системами), разрабатываются и поставляются в рамках настоящего
  технического задания.

### 5.4.1 Функциональные требования к сервису

Сервис должен обеспечивать выполнение следующего набора функций:

- инициирование проверки:
  o возможность отправки поискового запроса из интерфейса мессенджера в Систему.
  o поддержка отправки запросов в формате текстового сообщения (свободный ввод). Состав поддерживаемых форматов подлежит
  уточнению на этапе технического проектирования.
- обработка запроса и поиск:
  o передача поискового запроса из мессенджера в API Системы.
  o выполнение поиска по массивам данных, содержащихся в Поисковой платформе.
- формирование и доставка результата (шаблоны должны быть включены в технический проект):
  o автоматическое формирование структурированного ответа по результатам поиска.
  o отправка результата обратно в чат с пользователем в мессенджере в формате текстового сообщения.
  o в результатах выдачи должны быть указаны:
  o краткая информация о найденных аналогах/дубликатах (номер документа, название, статус, правообладатель).
  o ссылка (URL) на полную карточку документа в веб-интерфейсе ИС «Поисковая платформа» для детального просмотра.
- управление диалогом:
  o реализация сценария поддержания диалога с пользователем для уточнения параметров запроса («Уточните класс МКТУ» или
  «Уточните дату») в случае, если исходный запрос является неполным или неоднозначным.

### 5.4.2 Требования к интеграции и архитектуре

- каналы взаимодействия: взаимодействие между мессенджером и ИС «Поисковая платформа» должно осуществляться через
  существующее или специально доработанное API Системы.
- протоколы и форматы: обмен данными должен производиться по защищенному протоколу HTTPS с использованием синхронных или
  асинхронных запросов.
- аутентификация и безопасность:
  o интеграция должна предусматривать механизм безопасной аутентификации запросов, поступающих от мессенджера (с
  использованием взаимной TLS-аутентификации).
  o обработка персональных данных пользователей мессенджера (при их передаче в составе запросов) должна осуществляться в
  соответствии с требованиями Федерального закона от 27.07.2006 № 152-ФЗ «О персональных данных».
- производительность: время обработки типового запроса от момента отправки сообщения пользователем до получения
  структурированного ответа в мессенджере не должно превышать 15 секунд (при штатной работе каналов связи и внутренних
  сервисов).

### 5.4.3 Требования к информационной безопасности

Исполнитель при разработке сервиса обязан руководствоваться общими требованиями к безопасности, установленными в п. 5.6
настоящего документа (Требования к Исполнителю), а также обеспечить:

- фильтрацию и валидацию входных данных, поступающих из мессенджера, для предотвращения внедрения вредоносного кода (
  SQL-инъекций, XSS-атак и пр.).
- логирование всех событий взаимодействия с мессенджером (время запроса, отправитель, статус обработки) в соответствии с
  требованиями к информационному обеспечению (п. 6.2).

### 5.4.4 Требования к документированию

В составе рабочей документации (согласно разделу 10) Исполнитель должен предоставить:

- описание сценариев использования сервиса;
- схему интеграционного взаимодействия между мессенджером и Системой;

- описание API методов, используемых для взаимодействия, включая форматы запросов/ответов и коды ошибок.
- руководство администратора по настройке и мониторингу интеграционного модуля.

### 5.4.5 Требования к разработке интерфейса взаимодействия с мессенджером

- Для обеспечения функционирования сервиса в мессенджере должен быть разработан и развернут программный компонент,
  реализующий интерфейс взаимодействия с пользователями (далее – интерфейсный модуль). Данный модуль разрабатывается и
  поставляется в составе сервиса.
- Интерфейсный модуль должен быть реализован в виде защищённого веб-приложения или иного компонента, поддерживающего
  интеграцию с API мессенджера. При наличии в мессенджере поддержки мини-приложений допускается их использование для
  расширенного ввода данных и визуализации результатов.
- Информационные сообщения и сообщения об ошибках, направляемые пользователю в мессенджер, должны соответствовать общим
  требованиям к эргономике, быть краткими, предельно ясными и не содержать специализированного программного кода или
  стек-трейсов.
- Приветственное сообщение (или сообщение-помощь) должно содержать краткую инструкцию по использованию сервиса и примеры
  формулировки запросов.
- Интерфейсный модуль должен принимать входящие сообщения от мессенджера через механизм веб-приложения или вызовов API
  мессенджера.
- Модуль должен обеспечивать проверку входящих запросов (верификация подписи, токена или сертификата в соответствии с
  требованиями мессенджера) для предотвращения несанкционированного доступа.
- Ответы пользователю должны отправляться с использованием API мессенджера. Поддерживаемые форматы - текстовые
  сообщения.
- Модуль должен поддерживать хранение контекста диалога для каждого пользователя (идентификатор пользователя
  мессенджера)
  на время, необходимое для завершения.
- Все обращения должны выполняться по протоколу HTTPS с использованием взаимной аутентификации или токенов доступа.
- Модуль должен вести логирование всех событий взаимодействия с мессенджером: время получения запроса, идентификатор
  отправителя, тип события, статус обработки, код ошибки (при наличии). Логи должны передаваться в централизованную
  систему сбора логов. В Системе должен быть спроектирован статистический отчет о событиях взаимодействия с
  меесенджером.
- В составе рабочей документации должна поставляться схема развертывания интерфейсного модуля, инструкция по регистрации
  веб-приложения в мессенджере, описание используемых методов API мессенджера и примеры конфигурации.
